# SHA3-256 for Linux x86-64

SHA3-256 implemented as a handwritten x86-64 Keccak-f[1600] permutation using
AVX-512. The streaming hash API and the binary-safe command-line tool are both
written in assembly. The permutation uses `vprolq` for the Rho rotations and
`vpternlogq` for the three-input Theta and Chi logic, with all 25 state lanes
held one per vector register across the 32 EVEX registers.

The library requires x86-64 Linux hardware with AVX-512F and AVX-512VL (AMD
Zen 4 or later, Intel server cores); there is no runtime dispatch or baseline
x86-64 fallback. The command-line tool is a freestanding static binary that
uses raw Linux syscalls and no libc.

## Directory structure

- `src/sha3-256.s` implements the streaming API and Keccak permutation.
- `src/sha3-256.h` defines the C-compatible API and opaque context.
- `src/main.s` implements the stdin command-line tool.
- `test/` contains API and CLI correctness tests.
- `bench/` contains the optional OpenSSL/libgcrypt benchmark.
- `bin/` and `build/` are generated by the Makefile.

## Build and test

The normal build only needs gcc and GNU make:

```sh
make
make test
```

`make` creates the static x86-64 ELF executable at `bin/sha3-256`. `make test`
checks published SHA3-256 vectors, rate-boundary and irregular chunking,
independent contexts, context/output bounds, large binary stdin, and CLI error
handling.

## Command line

The command hashes standard input and writes a lowercase 64-character digest
followed by a newline:

```sh
printf abc | bin/sha3-256
# 3a985da74fe225b2045c172d6bd390bd855f086e3e9d525b46bfe24511431532

bin/sha3-256 < file.dat
```

The tool accepts no operands. It reads stdin in 64 KiB chunks and is binary
safe.

## C API

The public API consists of a caller-owned context and three functions:

```c
#include "src/sha3-256.h"

sha3_256_ctx context;
uint8_t digest[SHA3_256_DIGEST_BYTES];

sha3_256_init(&context);
sha3_256_update(&context, first_chunk, first_length);
sha3_256_update(&context, second_chunk, second_length);
sha3_256_digest(&context, digest);
```

The 344-byte context contains the 200-byte Keccak state, a 136-byte pending
rate block, and its eight-byte length. Independent contexts can be used
concurrently. `sha3_256_digest` finalizes destructively; call
`sha3_256_init` before reusing a finalized context. A null data pointer is
accepted by `sha3_256_update` only when the length is zero. Pointer validation
and finalized-state tracking are intentionally outside this minimal API.

## Implementation

Complete 136-byte blocks are absorbed directly from the caller's input. Only a
partial final block is copied into the context. During bulk absorption all 25
64-bit state lanes remain in vector registers across blocks, one lane per
register in `%xmm0`-`%xmm24` with `%xmm25`-`%xmm31` as scratch. Keccak rounds
are expanded four at a time in a six-iteration loop, balancing loop overhead
and code size. Each round is a 111-instruction vector schedule: `vpternlogq`
accumulates the Theta column parities three lanes at a time, the Theta
correction and Rho rotation are fused and walk the Pi permutation cycle so
every result lands directly in its canonical register, and Chi runs in place
per row as five more `vpternlogq`. Input lanes are absorbed with eight-byte
broadcast XORs that never read past the block. Context clearing, pending-block
copies, and final padding use 32- and 16-byte vector operations with bounded
byte tails.

Finalization uses the SHA3 domain suffix `0x06`, sets the last rate byte's
`0x80` bit, absorbs the padded block, and returns the first 32 little-endian
state bytes.

## Benchmark

The optional benchmark requires OpenSSL 3 and libgcrypt development headers
discoverable through `pkg-config` (`openssl-devel` and `libgcrypt-devel` on
Fedora):

```sh
make bench
bin/bench
```

It reuses allocated contexts and buffers, verifies that assembly, OpenSSL, and
libgcrypt produce identical digests at every size, and then times each hash
lifecycle. Every measurement first warms the core for at least 200 ms, then
reports the fastest of five equal repetitions; scheduling and frequency noise
only ever slow a run down, so the fastest repetition is the most repeatable.
The final line reports the worst spread between the fastest and slowest
repetition.

The following run was measured on an AMD EPYC 9575F (Zen 5) inside a KVM
guest with two vCPUs; virtualization makes the repetition spread larger than
on bare metal. The last two columns are reference time divided by assembly
