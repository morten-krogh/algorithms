# SHA3-256 for macOS ARM64

SHA3-256 implemented as a handwritten ARM64 Keccak-f[1600] permutation for
Apple Silicon. The streaming hash API and the binary-safe command-line tool are
both written in assembly. The permutation uses the ARM SHA3 extension
instructions `eor3`, `rax1`, `xar`, and `bcax`.

The library requires ARM64 macOS hardware with `FEAT_SHA3`; there is no runtime
dispatch or baseline ARM64 fallback. The library and command-line tool target
macOS 11 or later.

## Directory structure

- `src/sha3-256.s` implements the streaming API and Keccak permutation.
- `src/sha3-256.h` defines the C-compatible API and opaque context.
- `src/main.s` implements the stdin command-line tool.
- `test/` contains API and CLI correctness tests.
- `bench/` contains the optional CryptoKit/OpenSSL benchmark.
- `bin/` and `build/` are generated by the Makefile.

## Build and test

The normal build only needs Xcode Command Line Tools:

```sh
make
make test
```

`make` creates the ARM64 Mach-O executable at `bin/sha3-256`. `make test`
checks published SHA3-256 vectors, rate-boundary and irregular chunking,
independent contexts, context/output bounds, large binary stdin, and CLI error
handling.

## Command line

The command hashes standard input and writes a lowercase 64-character digest
followed by a newline:

```sh
printf abc | bin/sha3-256
# 3a985da74fe225b2045c172d6bd390bd855f086e3e9d525b46bfe24511431532

bin/sha3-256 < file.dat
```

The tool accepts no operands. It reads stdin in 64 KiB chunks and is binary
safe.

## C API

The public API consists of a caller-owned context and three functions:

```c
#include "src/sha3-256.h"

sha3_256_ctx context;
uint8_t digest[SHA3_256_DIGEST_BYTES];

sha3_256_init(&context);
sha3_256_update(&context, first_chunk, first_length);
sha3_256_update(&context, second_chunk, second_length);
sha3_256_digest(&context, digest);
```

The 344-byte context contains the 200-byte Keccak state, a 136-byte pending
rate block, and its eight-byte length. Independent contexts can be used
concurrently. `sha3_256_digest` finalizes destructively; call
`sha3_256_init` before reusing a finalized context. A null data pointer is
accepted by `sha3_256_update` only when the length is zero. Pointer validation
and finalized-state tracking are intentionally outside this minimal API.

## Implementation

Complete 136-byte blocks are absorbed directly from the caller's input. Only a
partial final block is copied into the context. During bulk absorption all 25
64-bit state lanes remain in vector registers across blocks. Keccak rounds are
expanded four at a time in a six-iteration loop, balancing loop overhead and
code size. Each round uses a 67-instruction vector schedule: `xar` fuses the
Theta correction with Rho rotation, and Chi consumes the resulting
Pi-permuted register layout directly before restoring the canonical layout.
State loads and stores are paired. Context clearing, pending-block copies, and
final padding use 32- and 16-byte vector operations with bounded byte tails.

Finalization uses the SHA3 domain suffix `0x06`, sets the last rate byte's
`0x80` bit, absorbs the padded block, and returns the first 32 little-endian
state bytes.

## Benchmark

The optional benchmark requires macOS 26 for `CryptoKit.SHA3_256`, Swift, and
OpenSSL 3 discoverable through `pkg-config`:

```sh
make bench
bin/bench
```

It reuses allocated contexts and buffers, verifies that assembly, CryptoKit,
and OpenSSL produce identical digests at every size, and then times each hash
lifecycle. The following run was measured on an Apple M1 Max. The last two
columns are reference time divided by assembly time, so values above 1.0 mean
the assembly implementation is faster.
